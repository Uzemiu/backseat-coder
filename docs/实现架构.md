# 实现架构

本文说明能工智人**怎么实现**：整体架构、双层职责划分、大脑层接口、以及 VS Code 插件形态的技术方案。产品层面见 [产品定义.md](产品定义.md)，原则层面见 [设计原则.md](设计原则.md)，落地排期见 [黑客松一天计划.md](黑客松一天计划.md)。

> **实现状态（截至当前）**：VS Code 侧边栏插件已落地，保命层与加分层全部完成——扫描、任务导航、改动审视、学习日志、编辑感知、AFK 静默均可用。唯一未做的是犹豫感知（4.5，规划中的增量项）。
>
> **一处关键的架构升级**：原计划插件通过 HTTP 调 `localhost:3000` 的 Node 服务，实现时改成了**插件进程内直接调用大脑层**（`require("./src/app")` 拿到 handlers），不再需要先手动 `npm run dev` 起服务。HTTP server（`server.js`）保留下来，作为浏览器 / 独立模式的备用入口。

## 一、双层架构

```
┌─────────────────────────────────────┐        ┌──────────────────────────────────┐
│  感知层 · VS Code 插件                 │        │  大脑层 · Node 模块（同进程 require）  │
│  extension.js + src/webview.js        │        │  src/app.js createHandlers(deps)   │
│                                       │  进程   │  · scan       项目理解              │
│  · 侧边栏 Webview（卡片流 UI）          │  内直调  │  · navigate   任务导航              │
│  · 编辑行为感知（敲键 / 光标 / 保存）     │ ─────► │  · diff       改动审视              │
│  · 在场检测（AFK / 失焦）               │ ◄───── │  · diffCheck  轮询式增量审视         │
│  · 把反馈渲染成侧边栏卡片                │  函数   │  · saveSession/getSessions 学习日志 │
│  （犹豫感知：规划中，暂未实现）            │  调用   │  · clearSessions 清除记忆           │
└─────────────────────────────────────┘        │  · AI provider 抽象 + 本地兜底        │
                                                 │  · 本地记忆（data/sessions.json）    │
                                                 └──────────────────────────────────┘

（浏览器 / 独立模式：server.js 把同一套 handlers 包成 HTTP 路由，
  前端 public/app.js 的 api() 同时兼容 webview postMessage 与 fetch 两种调用）
```

**核心思想：插件不重写任何业务逻辑，只做"感知 + 呈现"，直接复用大脑层的 handlers。**

这是这个项目能快速落地的根本原因——大脑层已经写好并测试过了，插件只需要把"过程信息"喂给它、把它的输出渲染出来。相比原计划的 HTTP 方案，进程内直调还省掉了"demo 前必须先起服务"这个最大的翻车风险点。

## 二、大脑层：可复用的 handlers

大脑层的能力集中在 `src/app.js` 的 `createHandlers(deps)`，返回一组 async 函数。插件和 HTTP server 都调用**同一组 handler**，只是入口不同：

| handler | 输入 | 输出 | 对应能力 | HTTP 路由（独立模式） |
|---------|------|------|---------|---------------------|
| `scan` | `{ repoPath }` | `{ tree, keyFiles, files, repoMap, aiSource }` | 项目理解 | `POST /api/scan` |
| `navigate` | `{ repoPath, task, repoMap, files }` | `{ searchResults, guide, learningMemory, aiSource }` | 任务导航 | `POST /api/navigate` |
| `diff` | `{ repoPath, repoMap }` | `{ diff, changedFiles, coach, learningMemory, aiSource }` | 改动审视（手动，brief+详情一次给全） | `POST /api/diff` |
| `diffDetails` | `{ repoPath, repoMap, diff, changedFiles }` | `{ coach, learningMemory, aiSource }` | 按需生成详情（点"展开详情"时） | `POST /api/diff/details` |
| `diffCheck` | `{ repoPath, repoMap, previousDiffHash, dirtyBuffers }` | `{ changed, hasDiff, worthMentioning, diffHash, coach, ... }` | 轮询式增量审视（只出 brief） | `POST /api/diff/check` |
| `explainCode` | `{ repoPath, repoMap, filePath, code, symbolName, lineNumber }` | `{ explanation, filePath, symbolName, aiSource }` | 阅读陪伴：讲解光标所在代码 | `POST /api/explain` |
| `saveSession` | 会话对象 | 保存后的会话 | 保存学习日志 | `POST /api/sessions` |
| `getSessions` | — | 会话数组 | 读取学习日志 | `GET /api/sessions` |
| `clearSessions` | — | `{ cleared: true }` | 清除该仓库记忆 | `POST /api/clearSessions` |

**关键 handler `diffCheck`**：它先算 diff 的 sha256 哈希，只有当哈希变了（`changed: true`）才真正调用 AI。这是插件"轮询感知改动但不烧 token"的现成机制——插件带上上次的 `diffHash` 调它即可。插件里由保存、修改、定时轮询三个触发源共用一个重入锁，保证同一份 diff 不被重复推成多张卡片（见 `extension.js` 的 `runDiffCheck`）。

**未保存缓冲区审视（`dirtyBuffers`）**：`git diff` 只看得见磁盘，看不到编辑器里还没保存的内容。为了让反馈"不保存就出现"，插件把当前 dirty 的缓冲区（`{ relpath, content }[]`）一并传给 `diffCheck`。大脑层的 `gitService.getDiffWithBuffers` 对每个缓冲区取其 HEAD 内容，用 `git diff --no-index --ignore-cr-at-eol` 对比临时文件生成 unified diff（零新依赖、`--ignore-cr-at-eol` 规避 Windows 上 CRLF 误报），再和磁盘 diff 合并（同一文件缓冲区版本优先）。`aiDiffCoach` 只吃 diff 文本、不关心来源，所以大脑层其余逻辑零改动。无 `dirtyBuffers` 时（浏览器/独立模式）退化为原来的 `getDiff`。

**懂分寸（`worthMentioning`）**：既然连未保存的半成品都会触发检查，就更要克制。`aiDiffBrief` 的返回含 `worthMentioning` 布尔字段，让 AI 自己判断"这次改动值不值得打断你"——只对真正空洞的改动（纯占位/空白/格式）保持沉默，其余有实质内容的改动都会开口。插件只在 `worthMentioning` 为真时推卡片，但无论如何都更新 `lastDiffHash`（避免对同一内容反复评估）。本地启发式兜底默认 `true` 保持现状不退化。

**brief / details 两段式（提速）**：AI 往返是主要延迟来源。感知阶段的 `diffCheck` 只调 `aiDiffBrief`——只生成 `worthMentioning` + `brief` + `summary` 三个小字段，输出小、返回快，让卡片第一时间冒出来。较重的 `impact/risks/missingTests/developerUnderstandingQuestions` 拆到 `aiDiffDetails`，仅当开发者点侧边栏卡片上的"展开详情"时，前端带上**卡片当时的 diff 快照**调 `diffDetails` 按需生成（用快照而非重读磁盘，保证详情对应的正是那张卡片的改动）。手动 `diff` handler 则 brief+详情一次给全。

> **注意 `getDiff` 会 `git add -N`**：为了让未跟踪的新文件也出现在 diff 里，`src/services/git.js` 用 intent-to-add 记录其存在（不暂存内容）。这是有意行为，不是副作用。

## 三、感知层：插件已实现的部分

以下能力都已在 `extension.js` + `src/webview.js` 落地。标注 ✅ 的已完成，🔴 的仍是规划中。

### 3.1 骨架（✅ 已完成）

- `package.json` 声明贡献点：`viewsContainers.activitybar` 一个侧边栏容器 + `views` 一个 `WebviewView`（`backseatCoder.sidebar`）
- 一条命令 `backseat-coder.open` 聚焦侧边栏
- F5 启动扩展开发宿主即可调试（`.vscode/launch.json`）

### 3.2 侧边栏 Webview（✅ 已完成）

- `vscode.window.registerWebviewViewProvider` 注册侧边栏面板（`src/webview.js`）
- Webview 复用 `public/` 里的 `index.html` / `app.js` / `styles.css`，渲染"卡片流"——每条反馈是一张卡片，往下追加
- `getWebviewHtml` 把静态资源路径改写成 `webview.asWebviewUri`，一套前端浏览器和插件两用

### 3.3 插件 ↔ Webview 通信（✅ 已完成）

- Webview 与插件主进程通过 `postMessage` 双向通信
- 消息协议：
  - 插件 → Webview：`{ type:'card', kind:'diff'|'scan'|'navigate', payload }`、`{ type:'afk' }`、`{ type:'wake' }`、`{ type:'init', workspacePath }`
  - Webview → 插件：`{ command, payload }`（API 调用，带数字 `id` 做请求/响应配对）、`{ command:'setContext', payload }`（同步当前上下文）、`{ command:'ready' }`
- 前端 `api()` 函数把 `/api/xxx` 风格调用透明映射成 postMessage，浏览器下则退化为 `fetch`
- 卡片流用 `retainContextWhenHidden` + `vscode.setState/getState` 持久化，面板隐藏后不丢历史

### 3.4 编辑行为感知（✅ 已完成）

- `vscode.workspace.onDidSaveTextDocument` — 保存后（防抖 `SAVE_DEBOUNCE_MS = 1500ms`）触发 diff 审视
- `vscode.workspace.onDidChangeTextDocument` — 敲键/删改（防抖 `CHANGE_DEBOUNCE_MS = 2500ms`，等你真正停手再检查，避免还在打字就触发）
- 每次事件都 `markActivity()` 更新最后活动时间戳
- 事件先按"文件是否在当前 repo 路径下"过滤，再进防抖
- `collectDirtyBuffers()` 采集 repo 内所有未保存缓冲区（`doc.isDirty` 且 `scheme==='file'`），随每次检查传给 `diffCheck`，实现"不保存也反馈"
- **触发源分级节流**：`runDiffCheck(trigger)` 区分 `save`/`change`/`poll`。`save` 是明确检查点，任何时候都跑；`change`/`poll` 距上次检查不足 `MIN_UNSAVED_CHECK_INTERVAL_MS = 5s` 就跳过，避免打字过程中反复烧 token（配合 `worthMentioning` 与 diffHash 去重，共同满足原则六懂分寸）

### 3.5 在场检测 AFK（✅ 已完成）

- 记录 `lastActivityAt`；定时器 `tickAfk`（每 30s）判断空闲超阈值（`AFK_TIMEOUT_MS = 2 分钟`）→ 进入静默
- `vscode.window.onDidChangeWindowState` — 编辑器失焦（切走）时立即静默
- 静默期间 `runDiffCheck` 直接返回，不轮询、不加卡片
- 恢复活动 → `wakeUp()`，向侧边栏推 `wake` 卡片（"欢迎回来"）
- 另有 `POLL_INTERVAL_MS = 30s` 定时轮询兜底：即便没有保存/修改事件也周期性查 diff，且 AFK 时自行跳过
- 光标移动（`onDidChangeTextEditorSelection`）也算 `markActivity`，避免你只读代码不编辑时被误判 AFK

### 3.6 犹豫感知（🔴 规划中，尚未实现）

- 计划从编辑事件流里用启发式判断"卡住了"：同一段代码短时间内反复删改、光标在少数几行里长时间徘徊、长时间停在一处不动
- **难点是调参**：什么模式算犹豫、阈值多少，没有标准答案
- **降级策略**：即使判断不完美，侧边栏里偶尔来一张不太准的卡片，远没有正文误弹刺眼，容错率高
- 属于"有余力才做"的增量，主线不依赖它——当前版本主线已完整，这一项留待后续

### 3.7 阅读陪伴 / 光标讲解（✅ 已完成）

- `vscode.window.onDidChangeTextEditorSelection` — 光标移动后（防抖 `EXPLAIN_DEBOUNCE_MS = 4000ms`，等你在一处停下）触发 `runExplain`
- **取讲解范围**：`findEnclosingSymbol` 用 `vscode.executeDocumentSymbolProvider` 找包含光标行的最内层函数/方法/类，取其整段源码；拿不到符号（无语言服务）时回退到光标上下各 20 行
- 调 `explainCode` handler → `aiExplainCode`，返回 `{ brief, role, watchOut }`，推 `kind:'explain'` 卡片
- **护栏（省 token + 懂分寸）**：AFK 跳过；距上次编辑不足 `EXPLAIN_MIN_IDLE_AFTER_EDIT_MS = 1.5s` 跳过（那是在写不是在读）；同一函数（`lastExplainKey`）不重复讲；重入锁 `explainInFlight`；范围 <20 或 >6000 字符不讲（没内容 / 成本高且多半选错范围）
- **只解释不建议改**：prompt 明确要求这是阅读辅助，narrate 代码意图与角色，不做 review、不提改动建议（原则一 引导而非代劳）

## 四、数据流示例

**场景：你在 `auth.js` 里改代码（还没保存）**

```
1. onDidChangeTextDocument 触发 → markActivity()（防抖 2.5s，等你停手）
2. runDiffCheck("change")：距上次检查够 5s 才继续（否则跳过省 token）
3. collectDirtyBuffers() 采集 auth.js 的未保存内容 → 传给 handlers.diffCheck
4. getDiffWithBuffers 用 git diff --no-index 对比 HEAD 与缓冲区 → 合成 diff
5. diff 哈希变了 → 调 AI 生成 coach，含 worthMentioning 判断
6. worthMentioning=true 且 changed → postMessage { type:'card', kind:'diff' }
   worthMentioning=false → 只更新 lastDiffHash，不推卡片（安静）
7.（保存时走 runDiffCheck("save")，防抖 1.5s，且不受 5s 间隔限制，反馈更快）
```

**场景：你去茶歇了**

```
1. 插件定时器 tickAfk 发现 2 分钟无编辑活动（或窗口失焦立即触发）
2. goAfk() → 停止后续 runDiffCheck（内部检查 isAfk 直接返回）
3. postMessage：{ type:'afk' } → 侧边栏显示"💤 已静默"
4. 你回来敲键 → onDidChangeTextDocument → markActivity() → wakeUp()
5. postMessage：{ type:'wake' } → 侧边栏显示"👋 欢迎回来"
```

## 五、技术选型与约束

- **插件语言**：JavaScript（`extension.js` 为 CommonJS，无 TS 构建步骤，直接跑）
- **不引入重型前端框架**：webview 用原生 HTML/JS，符合[设计原则 3.1](设计原则.md)
- **大脑层零改动复用**：插件与 HTTP server 调用同一组 handlers；新行为遵循"`server.js` 保持薄、逻辑进 service"
- **无需先启动服务**：进程内直调，插件 F5 即用，不依赖 `localhost:3000`（HTTP server 仅浏览器/独立模式需要）
- **离线兜底继承**：大脑层本身在无 API Key 时用本地启发式输出，插件不用额外处理
