# 实现架构

本文说明能工智人**怎么实现**：整体架构、双层职责划分、现有后端接口、以及 VS Code 插件形态的技术方案。产品层面见 [产品定义.md](产品定义.md)，原则层面见 [设计原则.md](设计原则.md)，落地排期见 [黑客松一天计划.md](黑客松一天计划.md)。

---

## 一、双层架构

```
┌─────────────────────────────────────┐        ┌──────────────────────────────────┐
│  感知层 · VS Code 插件                 │        │  大脑层 · 本地 Node 服务（已存在）    │
│                                       │        │                                    │
│  · 侧边栏 Webview（卡片流 UI）          │  HTTP  │  · /api/scan     项目理解           │
│  · 编辑行为感知（敲键 / 光标 / 保存）     │ ─────► │  · /api/navigate 任务导航           │
│  · 在场检测（AFK / 失焦）               │ ◄───── │  · /api/diff     改动审视           │
│  · 犹豫启发式判断                       │  JSON  │  · /api/diff/check 轮询式增量审视    │
│  · 把反馈渲染成侧边栏卡片                │        │  · /api/sessions 学习日志读写        │
└─────────────────────────────────────┘        │  · AI provider 抽象 + 本地兜底        │
                                                 │  · 本地记忆（data/sessions.json）    │
                                                 └──────────────────────────────────┘
```

**核心思想：插件不重写任何业务逻辑，只做"感知 + 呈现"，把现有 Node 服务当后端调用。**

这是这个项目在一天内可落地的根本原因——大脑层已经写好并测试过了，插件只需要把"过程信息"喂给它、把它的输出渲染出来。

## 二、大脑层：现有后端接口

以下是当前 `src/app.js` 已实现、可直接调用的接口。**插件不需要改动这些，直接 `fetch` 即可。**

| 方法 | 路径 | 输入 | 输出 | 对应能力 |
|------|------|------|------|---------|
| POST | `/api/scan` | `{ repoPath }` | `{ tree, keyFileContents, repoMap, aiSource }` | 项目理解 |
| POST | `/api/navigate` | `{ repoPath, task, repoMap, files }` | `{ searchResults, guide, learningMemory, aiSource }` | 任务导航 |
| POST | `/api/diff` | `{ repoPath, repoMap }` | `{ diff, changedFiles, coach, learningMemory, aiSource }` | 改动审视 |
| POST | `/api/diff/check` | `{ repoPath, repoMap, previousDiffHash }` | `{ changed, hasDiff, diffHash, coach, ... }` | 轮询式增量审视 |
| POST | `/api/sessions` | 会话对象 | 保存后的会话 | 保存学习日志 |
| GET | `/api/sessions` | — | 会话数组 | 读取学习日志 |

**关键接口 `/api/diff/check`**：它先算 diff 的 sha256 哈希，只有当哈希变了（`changed: true`）才真正调用 AI。这是插件"轮询感知改动但不烧 token"的现成机制——插件定时带上上次的 `diffHash` 调它即可。

## 三、感知层：插件要从零做的部分

### 3.1 骨架（🟢 低难度，~1-2h）

- `yo code` 生成 TypeScript 插件项目
- `package.json` 声明贡献点：一个侧边栏容器 + 一个 `WebviewView`
- F5 启动扩展开发宿主调试

### 3.2 侧边栏 Webview（🟢 低难度，~2-3h）

- 用 `vscode.window.registerWebviewViewProvider` 注册侧边栏面板
- Webview 里是一段 HTML/JS，渲染"卡片流"——每条反馈是一张卡片，往下追加
- 现有 `public/` 里的前端展示逻辑可大量借用

### 3.3 插件 ↔ Webview 通信（🟡 中难度）

- Webview 是独立沙箱，与插件主进程通过 `postMessage` 双向通信
- 约定消息协议，例如：
  - 插件 → Webview：`{ type: 'card', kind: 'diff'|'hesitation'|'afk'|'task', payload }`
  - Webview → 插件：`{ type: 'action', action: 'scan'|'navigate', payload }`
- 联调是这一层的主要耗时点（沙箱调试麻烦），不难但琐碎

### 3.4 编辑行为感知（🟡 中难度）

- `vscode.workspace.onDidChangeTextDocument` — 监听敲键/删改
- `vscode.window.onDidChangeTextEditorSelection` — 监听光标移动
- `vscode.workspace.onDidSaveTextDocument` — 保存时触发 diff 审视
- **注意**：这些事件极其频繁，必须做防抖/节流，否则既卡又烧 token

### 3.5 在场检测 AFK（🟡 中难度）

- 记录"最后一次编辑活动时间戳"
- 定时器判断：超过阈值（比如 2 分钟）无活动 → 进入静默
- `vscode.window.onDidChangeWindowState` — 编辑器失焦（切走）时也进入静默
- 静默期间：停止轮询 `/api/diff/check`，不加新卡片
- 恢复活动 → 唤醒，可选地加一张"欢迎回来，你之前在改 X"的卡片

### 3.6 犹豫感知（🔴 高不确定，增量做）

- 从编辑事件流里用启发式判断"卡住了"：同一段代码短时间内反复删改、光标在少数几行里长时间徘徊、长时间停在一处不动
- **难点是调参**：什么模式算犹豫、阈值多少，没有标准答案，需要反复试
- **降级策略**：即使判断不完美，侧边栏里偶尔来一张不太准的卡片，远没有正文误弹刺眼，容错率高
- 一天时间里这是"有余力才做"的增量，不是主线

## 四、数据流示例

**场景：你保存了 `auth.js`**

```
1. 插件 onDidSaveTextDocument 触发（已节流）
2. 插件 fetch POST /api/diff/check { repoPath, previousDiffHash }
3. 大脑层算 diff 哈希 → 变了 → 调 AI 生成 coach 建议
4. 返回 { changed: true, coach: {...}, learningMemory: [...] }
5. 插件 postMessage 给 webview：{ type:'card', kind:'diff', payload: coach }
6. Webview 在卡片流末尾追加一张"改动审视"卡片
```

**场景：你去茶歇了**

```
1. 插件定时器发现 3 分钟无编辑活动
2. 插件停止 /api/diff/check 轮询
3. postMessage：{ type:'card', kind:'afk' } → 侧边栏显示"💤 已静默"
4. 你回来敲键 → onDidChangeTextDocument 触发 → 唤醒轮询
```

## 五、技术选型与约束

- **插件语言**：TypeScript（`yo code` 默认）
- **不引入重型前端框架**：webview 用原生 HTML/JS 即可，符合[设计原则 3.1](设计原则.md)
- **大脑层零改动优先**：一天时间内，除非必要不动后端；真要加，也遵循"`server.js` 保持薄、逻辑进 service"
- **本地服务需先启动**：插件假设 `http://localhost:3000` 已在跑（demo 时先 `npm run dev`）。有余力再让插件自动拉起服务
- **离线兜底继承**：大脑层本身在无 API Key 时用本地启发式输出，插件不用额外处理

## 六、风险点

| 风险 | 说明 | 缓解 |
|------|------|------|
| Webview 通信联调耗时 | 沙箱调试麻烦，最容易吃掉时间 | 先用最简协议打通一条链路，再扩展 |
| 编辑事件太频繁 | 不节流会卡、会烧 token | 一开始就上防抖，别等出问题 |
| 犹豫感知调不准 | 启发式没有标准答案 | 降级为增量功能，主线不依赖它 |
| 时间不够 | 一天很短 | 见 [黑客松一天计划.md](黑客松一天计划.md) 的砍功能优先级 |
