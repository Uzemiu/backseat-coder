# 产品定义

## 能工智人 · Backseat Driver

> 不是 AI 写、人来看，而是人来写、AI 来叨叨。

---

## 一、名称

**中文名：能工智人**

拆开来读，是两种身份的叠加：

- **能工**——技艺精湛的匠人，亲手打磨每一行代码
- **智人**——善用思维的学习者，真正理解自己做了什么

合在一起，描述的是一种工作状态：在 AI 的陪伴下，程序员自己成为真正掌握代码的人。

**英文名：Backseat Driver**

"Backseat driver"在英文里是个略带贬义的词，指那种坐在后排、不停给司机提意见却从不开车的人。把这个词当作项目名，是一种刻意的幽默和自嘲——对，这个 AI 就是那个坐在后座叨叨的家伙，但方向盘永远在你手里。

---

## 二、一句话定位

**一个不抢键盘的 AI Mentor，在旁边和你结对编程——看你在哪里犹豫、及时提醒哪里写错了、告诉你怎样能实现得更好，并且持续记住你的风格。**

主流 AI 编程工具的逻辑是：AI 写代码，人来审查。能工智人反过来：人来写代码，AI 坐在旁边看、读、提问、叨叨。

它不是一个你点一下才回应的分析工具，而是一个**有存在感的陪练**：

- **感知你的状态**——看到你在某处停留、反复修改，知道你卡住了
- **在合适的时机开口**——发现明显的错误或更好的实现方式，主动提醒，而不是等你来问
- **记住你这个人**——通过本地记忆持续磨合，越用越懂你的代码风格和习惯
- **知道你不在**——你去拉屎、茶歇的时候它会安静下来，不无谓地鞭策你，也不浪费 token

---

## 三、目标用户

- **接手陌生项目的开发者**：面对几万行不熟悉的代码，需要快速建立结构认知
- **对代码理解有要求的工程师**：不满足于接受 AI 生成的代码，想真正读懂自己维护的东西
- **需要交接与 Onboarding 的团队**：希望有可传递、可复用的项目知识

---

## 四、核心功能

> 下面 4.1–4.4 是当前 MVP 已具备的能力，4.5–4.7 是产品愿景中规划的"陪练"能力，标注为【规划中】。
>
> **需要强调的是：这些不是七个割裂的功能，而是一个陪练的不同侧面。** 产品目标是把它们融进**同一个无感的界面**——不需要你切标签页、点按钮，AI 在你写代码的过程中自然地感知、理解、适时开口。交互形态见[第五节](#五交互形态从功能分区到无感陪伴)。

### 4.1 理解项目 Repo Understanding

输入本地项目的绝对路径，能工智人扫描文件树、读取关键文件（README、package.json、CLAUDE.md 等），生成一份**项目导读**：

- 项目类型与技术栈判断
- 主要入口文件
- 核心模块划分
- 推荐阅读顺序
- 测试策略提示
- 需要人工确认的未知点

这份导读作为持久上下文注入后续所有 AI 请求，类似 CLAUDE.md 的作用。无 API Key 时用本地启发式逻辑兜底。

### 4.2 陪你上手任务 Task Navigation

你用自然语言说出要做的事（如"修改登录会话超时时间"），AI 搜索代码库，结合项目理解和历史记忆，给出：

- 优先阅读文件
- 可能需要修改的文件
- 动手前要回答的问题
- 建议的小步骤

AI 不写实现代码，只给路线图。

### 4.3 陪你审视改动 Change Review

你在编辑器里改代码的过程中，AI 读取 `git diff`，以导师而非评审官的语气帮你审视：

- 改动摘要
- 影响范围
- 潜在风险
- 缺失的测试
- 三个反思性问题

当前 MVP 通过**定时监视**实现：AI 在后台盯着 diff 变化，有变化才开口，diff 未变则不调用 AI、不消耗 token。愿景是让这份审视随你的编码节奏自然浮现，而不需要你主动去点"分析"。

### 4.4 学习日志与风格记忆 Learning Log & Style Memory

工作结束后保存会话记录：做了什么任务、改了哪些文件、学到了什么概念、还有哪些问题没有答案。存储在本地 `data/sessions.json`。

下次在同一仓库工作时，AI 自动检索最相关的历史记录作为项目记忆注入请求，UI 中明确展示哪些历史被召回。

**更进一步的愿景【规划中】**：本地记忆不只记录"做过什么"，还持续沉淀"你是怎么写代码的"——命名习惯、代码结构偏好、常用的库、反复出现的模式、乃至常犯的错。这些风格画像随时间积累，让 AI 的建议越来越贴合你本人，实现真正的**持续磨合**：用得越久，越懂你。所有记忆都存在本地，不上云。

### 4.5 犹豫感知 Hesitation Sensing【规划中】

一个真正在旁边的 mentor，能看出你什么时候卡住了。

通过编辑器扩展感知你的编辑行为——在某一处停留很久、反复删改同一段代码、光标在几个文件之间来回跳——判断你可能遇到了困难，然后在恰当的时机主动开口：

- "这个函数你改了好几遍了，是不是这里的边界条件不好处理？"
- "你在这两个文件之间来回看，需要我帮你理一下它们的调用关系吗？"

关键在于**时机与分寸**：只在你真正卡住时介入，不打断你顺畅的心流。

### 4.6 主动反馈 Proactive Feedback【规划中】

从"你点一下我才分析"变成"我发现问题主动提醒"。

在你编码的过程中，AI 持续在后台观察，发现两类情况时主动开口：

- **写错了**：明显的逻辑错误、拼写、类型不匹配、遗漏的空值处理——早提醒，而不是等到 Diff Coach 阶段
- **能更好地实现**：发现你在重复造轮子（项目里已有现成工具函数）、有更符合项目惯例的写法、有更简洁的实现方式

语气始终是 mentor 而非纠错机器：给出理由和更好的方向，而不是直接甩一段替换代码。

### 4.7 在场感知 Presence & AFK【规划中】

一个懂分寸的陪练，知道你什么时候不在。

当检测到你离开——一段时间没有键盘/编辑活动、锁屏、切换到其他应用——AI 自动进入**静默状态**：

- 不再触发主动反馈，不"鞭策"一个不在座位上的人
- 暂停定时分析，**不在你拉屎、茶歇、开会时浪费 token**
- 你回来（恢复编辑活动）后自动唤醒，必要时用一句话帮你重新进入状态："你离开前正在改 `auth.js` 的超时逻辑，还差一个测试。"

这条能力和"节制地调用 AI"的工程原则一脉相承：**在场才反馈，离开就安静。**

## 五、交互形态：从功能分区到无感陪伴

### 现状：功能被切成了标签页

当前 MVP 把能力分成了几个标签页——仓库地图、Navigator、Diff Coach、Learning Log——每个都要你手动切过去、填表单、点按钮。这是搭原型时最快的做法，但它和"一个在旁边陪你的 mentor"的定位是矛盾的：

> 没有哪个真正在旁边的搭档，会要求你"先切到第三个标签页、再点一下分析按钮"它才理你。

分区式的界面把负担推回给了用户——你得先知道自己需要哪个功能、切过去、把上下文（路径、任务、diff）再喂一遍。这本身就是一种打扰。

### 目标：一个界面，无感陪伴

产品的方向是把这些能力**收拢进同一个界面**，让它们围绕"你正在做的事"自然发生，而不是围绕"功能菜单"排列：

- **一条主线，而非多个入口**——界面中心是当前的工作上下文（在哪个项目、在做什么、改了什么），而不是一排功能标签
- **AI 主动浮现，而非等你调用**——项目理解、任务建议、改动审视、犹豫提醒，都在合适的时机自己出现在同一个地方，不需要你切换和触发
- **上下文自动衔接**——扫描、任务、diff、记忆共享同一份上下文，不需要在不同标签页之间反复输入
- **可被忽略，不可被强加**——所有浮现的内容都是可参考、可略过的旁注（原则三），它安静地待在视野边缘，需要时才展开

一句话：**让功能消失，让陪伴显现。** 用户感知到的不该是"我在用四个工具"，而是"有个 mentor 一直在旁边"。

### 与"懂分寸"的关系

无感不等于话多。恰恰相反——正因为要做到无感，才更需要[原则六「懂分寸」](设计原则.md)的约束：一个融进界面、随时可能开口的 AI，如果不懂时机，就会从"陪伴"退化成"骚扰"。无感陪伴的前提，是它知道什么时候该安静。

### 落地形态：VS Code 侧边栏插件

"无感陪伴"这个愿景，最终落到一个具体形态：**一个 VS Code 侧边栏插件，交互模型参照 Claude Code 的插件**。

为什么是侧边栏，而不是在代码正文里弹提示：

- **非侵入是形态自带的**——所有反馈都待在侧边的面板里，以卡片流的形式出现。它从不在你的代码正文里弹窗、也不抢占光标。你想看就扫一眼，不想看它就静静待着。这样"不打扰"这个属性靠形态本身就保证了，而不用靠精巧的时机算法。
- **一个面板容纳全部能力**——项目理解、当前任务、改动审视、犹豫提醒、AFK 状态，全都是同一个侧边栏里的内容，天然实现了"功能融成一体、不分 tab"。
- **贴着你的编辑器**——插件能感知你正在敲什么、光标停在哪、多久没动了，这些是当前 Web 应用形态拿不到的"过程信息"，也正是 mentor 愿景的关键。

形态示意：

```
VS Code 侧边栏（webview）
┌───────────────────────────┐
│ 能工智人 · 陪你写这个项目     │
├───────────────────────────┤
│ 📍 项目理解（可折叠）         │
│ 🧭 当前任务：改超时逻辑        │
├───────────────────────────┤
│ 反馈流（往下追加）：           │
│  · [diff] 你改了 auth.js…    │
│  · [犹豫] 这函数改了 4 遍…     │
│  · 💤 检测到你离开，已静默      │
└───────────────────────────┘
```

架构上，插件只承担"感知 + 呈现"，现有的 Node 服务作为"大脑"被复用。详见 [实现架构.md](实现架构.md)。

## 六、非目标

能工智人**不做**以下事情：

- 直接编辑或覆盖用户代码
- 实现复杂的 AST 级静态分析
- 替代正式的 Code Review
- 处理多人协作
- 依赖云端存储或身份认证
- 强制要求 API Key 才能运行

## 七、使用流程

无感陪伴不代表没有流程，而是让流程**贴着编码本身走**，而不是贴着功能菜单走：

```
指向一个项目（一次性）
  → 开始写代码
     ↓（以下全部在同一界面自然浮现，无需手动切换）
  · AI 在后台建立并持续更新对项目的理解
  · 你说出要做的事 → 上手建议随之出现
  · 你写代码时 → 卡住有提醒，写错有反馈，改动有审视
  · 你离开 → AI 静默；你回来 → 自动唤醒
  → 收工时理解自动沉淀为本地记忆，下次继续磨合
```

对比当前 MVP 的分步流程（扫描 → 切到 Navigator 输入任务 → 切到 Diff Coach 点分析 → 切到 Learning Log 保存），愿景形态里这些"切换"和"触发"都应消失。

---

## 八、典型使用场景

**接手陌生项目** — 用仓库地图在十分钟内建立对项目结构的初步认知，拿到推荐阅读顺序。

**完成一个具体需求** — 用任务导航定位相关文件，回答前置问题，带着理解去改，而不是边改边猜。

**提交前的最后检查** — 用改动教练确认影响范围清晰、测试没有遗漏、风险已评估。

**隔了很久回来维护** — 用学习日志唤起上次工作的上下文，不用从头把代码重读一遍。

**新人 Onboarding** — 让新同事自己用仓库地图和任务导航熟悉项目，减少对老员工的口头讲解占用。

## 九、AI Provider 支持

| 模式 | 说明 |
|------|------|
| `local` | 内置启发式逻辑，无需 API，可离线使用 |
| `anthropic` | Anthropic Messages API（推荐） |
| `openai` | OpenAI Chat Completions |
| `auto` | 有 Anthropic Key 用 Anthropic，有 OpenAI Key 用 OpenAI，否则 local |

无论哪种模式，Provider 调用失败时都会回退到本地兜底输出，保证应用始终可用。

## 十、技术栈概览

轻量本地 Node.js 应用（Node 20+），单一外部依赖（`undici`），无重型框架，无数据库，JSON 文件存储。详见 [设计原则](设计原则.md) 与仓库根目录的架构说明。
