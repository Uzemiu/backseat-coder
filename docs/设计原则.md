# 设计原则

本文说明能工智人**为什么**这样设计，以及在产品演进中坚持的核心原则。功能层面的定义见 [产品定义.md](产品定义.md)。

---

## 一、根本立场：颠覆 Vibe Coding

### 什么是 Vibe Coding

当前 AI 编程工具的主流范式：

```
人描述意图 → AI 生成代码 → 人接受/调整 → 继续
```

AI 写，人来看。Copilot、Cursor、Claude Code 都在这条路上越走越深。它的价值很真实——速度快、门槛低、适合原型和探索性工作。

### 它的隐性代价

Vibe Coding 提升了代码的生产速度，但悄悄带走了一些东西：

- AI 生成了一段实现，你不确定它为什么这样写
- 改动上线了，但你不清楚它影响了哪些模块
- 项目越来越大，只有 AI 的上下文窗口里还有完整图景——你自己的认知在退化
- 项目知识全在聊天记录里，换个工具、换个人，全部消失

**当 Vibe Coding 成为唯一的工作方式，程序员与代码库的连接就断了。**

### 能工智人的反转

```
人写代码 → AI 陪在旁边读、看、叨叨 → 人理解、决策、提交
```

人来写，AI 来看。这不是否定 Vibe Coding，而是回答一个不同的问题：**当你需要真正读懂代码库、对改动负责、把知识留下来的时候，你需要的不是一个帮你写代码的工具。**

## 二、核心原则

### 原则一：引导，而非代劳

所有功能的设计都服务于一个铁律：**AI 的输出是方向和问题，不是最终代码。**

- AI 告诉你该读哪些文件，但你自己去读
- AI 提出改动前应该回答的问题，但你自己去找答案
- AI 分析改动的风险，但你自己决定怎么处理

这条原则直接约束了功能设计：任务导航返回"路线图"而非实现；改动教练返回"反思问题"而非修改建议。**一旦 AI 开始替人写代码，这个产品就失去了存在的意义。**

### 原则二：理解优先于速度

能工智人不追求让你更快地产出代码，它追求让你在完成工作的同时真正理解你做了什么。

一个真正读懂代码库的程序员，能做出更好的架构决策，能更快定位 bug，能更安全地重构，能把知识传递给下一个人。这比多生成一百行代码更有价值。

**设计含义**：宁可让 AI 少说、多问，也不要让它给出一个让人不假思索就接受的答案。

### 原则三：程序员始终在控制席

项目代号"backseat-coder"——副驾座上的人。副驾可以看地图、指路、提醒危险，但方向盘始终在司机手里。

**设计含义**：任何时候都不修改用户的代码；所有产出都是可参考、可忽略的建议，而非需要"接受/拒绝"的补丁。

### 原则四：知识沉淀是一等公民

每次工作都会产生理解——弄清楚了某个模块的逻辑、踩了某个坑、发现了某个隐藏依赖。这些东西不应该消散在聊天记录里。

**设计含义**：学习日志不是可选的附属功能，而是核心闭环的一环。它把一次性的理解转化为可被后续 AI 请求检索的持久记忆，并在 UI 中透明展示被召回的记录，让用户看见"这次建议基于什么上下文"。

### 原则五：真正的 Pair Coding

结对编程的原始定义里有两个角色：Driver 掌键盘写代码，Navigator 保持全局视角、提问题、发现盲点。

能工智人实现的是这个关系的 AI 版本：

- **你是 Driver**：键盘在你手里，代码由你来写
- **AI 是 Navigator**：读懂代码库，引导方向，在你动手前提问，在你改完后帮你审视盲点

两个人都在写代码，不叫结对编程。一个人写、另一个人保持全局视角并持续提问，才是。

### 原则六：懂分寸——在场才反馈，离开就安静

一个好的 mentor 知道什么时候该开口，什么时候该闭嘴。一个坐在旁边不停鞭策、你上个厕所回来还在唠叨的陪练，只会让人想把它关掉。

能工智人追求的是**恰到好处的介入**：

- 你顺畅编码时，它安静观察，不打断心流
- 你卡住、写错、有更好写法时，它才在合适时机开口
- 你离开座位（AFK）时，它进入静默——不反馈、不分析、不浪费 token
- 你回来时，它自动唤醒，必要时帮你重新进入状态

**设计含义**：主动反馈能力必须内建"时机判断"和"在场检测"，宁可少说也不要在错误的时刻打扰。**分寸感不是可选的礼貌，而是这类主动型 AI 能否被长期使用的生死线。**

### 原则七：越用越懂你

AI mentor 的价值随时间增长。第一天它只懂这个代码库，用了一个月后，它应该懂**你**——你的命名习惯、结构偏好、常用的库、反复踩的坑。

**设计含义**：本地记忆不止记录"做过什么任务"，还持续沉淀开发者的风格画像，并将其注入后续建议，实现真正的持续磨合。所有个人风格数据都留在本地，不上云，这既是隐私底线，也是"这台机器上的 AI 真正属于你"的体现。

### 原则八：让功能消失，让陪伴显现

一个真正在旁边的搭档，不会要求你"先切到第三个标签页、再点一下按钮"它才理你。用功能菜单组织界面，等于把"我该用哪个功能、去哪里触发它"的负担推回给用户——这本身就是一种打扰。

能工智人的界面应该围绕**你正在做的事**组织，而不是围绕**功能清单**组织：

- 一条主线（当前项目 / 当前任务 / 当前改动），而非一排标签页
- AI 的洞察在合适时机自己浮现，而非等你切换过去手动调用
- 上下文在各能力之间自动衔接，而非在不同界面里反复输入

**设计含义**：新能力优先考虑"如何融进已有的主界面并在恰当时机浮现"，而不是"新开一个标签页/面板"。判断标准是：用户感知到的应该是"有个 mentor 一直在旁边"，而不是"我在同时用好几个工具"。

**与原则六的张力**：无感 ≠ 话多。越是要把 AI 融进界面、让它随时可能开口，就越依赖[原则六「懂分寸」]的约束。无感陪伴的前提，是它知道什么时候该安静。这两条原则必须一起成立——只做无感不做分寸，就是骚扰。

### 原则九：反馈归拢侧边栏，非侵入优先

所有 AI 反馈都待在**侧边栏面板**里，以卡片流的形式出现——绝不在代码正文里弹窗、绝不抢占光标、绝不打断你正在敲的那一行。

这不只是 UI 偏好，而是一个降低"打扰成本"的根本决定：

- 侧边栏天然是非侵入的。反馈就在那儿，你扫一眼就知道有没有新东西，不看它也不会挡住任何代码。
- "不打扰"这个属性由**形态本身**保证，而不用靠精巧的"该不该弹"时机算法（那是个无底洞，也最容易把陪伴做成骚扰）。
- 一个面板容纳全部能力，天然实现原则八的"功能融成一体"。

**设计含义**：不做 inline decoration / hover 弹框 / 打断式通知。要说的话，往侧边栏的流里加一张卡片；控制的是"加卡片的频率"（节流 + AFK 静默），而不是"弹框弹不弹"。

## 三、工程设计原则

### 3.1 保持简单，可离线运行

- 轻量本地 Node.js 应用，单一外部依赖（`undici`），无重型框架、无数据库
- JSON 文件存储会话数据
- **API Key 非必须**：本地启发式模式完整可用，保证演示可靠、离线可用
- 所有 AI 响应都有本地兜底，Provider 调用失败不会导致应用崩溃

### 3.2 稳定的上下文注入

扫描生成的 `projectGuide` 会注入扫描之后的每一个 AI 请求，让模型始终获得稳定的仓库上下文——类似工具使用 CLAUDE.md 的方式。这确保 AI 的建议始终扎根于真实项目，而非泛泛而谈。

### 3.3 有意识的资源限制

仓库扫描刻意限制了扫描深度、文件数量、文本文件大小和 AI Prompt 大小。这些限制是产品行为的一部分，不应在无意间被移除。

### 3.4 节制地调用 AI

定时监视先计算 diff 哈希，只有当 diff 真正变化时才调用 AI Provider。这让"持续监视"这个功能在不反复烧 token 的前提下保持有用。

### 3.5 安全与日志

- 结构化 JSON 日志，记录每一个重要操作，便于排查
- 日志中自动脱敏密钥、令牌、密码、Authorization 头等敏感信息
- `.env` 可能包含真实密钥，绝不提交、绝不完整打印

### 3.6 感知层与大脑层分离

产品形态是 VS Code 插件，但插件只承担"感知 + 呈现"这一层——捕获编辑行为、检测在场、把反馈渲染进侧边栏。所有"大脑"逻辑（项目理解、任务导航、diff 分析、本地记忆、AI provider 抽象与本地兜底）仍然由现有的 Node 服务承担。

这样分层的好处：

- 现有后端代码（6 个 `/api` 路由）**直接复用**，插件不重写业务逻辑，只 `fetch` 本地服务
- 大脑层与形态解耦，将来出 JetBrains 插件、CLI 版本，大脑不用重写
- 大脑层可独立测试、可离线兜底，不被 UI 形态绑架

详见 [实现架构.md](实现架构.md)。

## 四、演进方向上的原则

产品会演进，但以下原则不变：

- **小而兼容的改动优先**：这是一个前后端契约简单的本地 MVP，改 API 响应结构就要同步改前端
- **`server.js` 保持薄**：新行为放进 service 或 `app.js` 编排层
- **不引入重型框架**：除非有明确的迁移决策
- **保留本地兜底**：无论 Provider 层如何变化，应用必须在没有模型 API 时依然可用
- **界面向"无感陪伴"收敛**：当前的多标签页形态是原型期的权宜，演进方向是把能力融进单一主界面（原则八），而非继续堆叠新标签页

从 Guide（新人向导）到 Navigator（任务导航）再到 Reviewer（改动评审），AI 的角色可以随着开发者对项目的熟悉程度而演进——但它永远不会从后座挪到驾驶座。

## 五、一句话总结

Vibe Coding 让代码变得容易生成。能工智人让代码变得容易理解。

> 坐在后座，看你开车，该叨叨的时候叨叨，该闭嘴的时候闭嘴。方向盘永远是你的。
